# 项目记忆

最后更新：2026-07-17

## 项目目标

“我的哑铃教练”是一个面向力量训练新手的个人健康助手。核心体验不是堆叠功能，而是让用户每天打开后立即知道：今天练什么、身体是否适合正常训练、饮食和睡眠还缺什么，以及坚持了多久。

产品原则：

- 新手优先：动作少、反馈直接、避免复杂健身术语。
- 本地优先：个人健康数据默认不离开设备。
- 用户确认：食物识别和营养数据必须明确为估算值，并允许修改。
- 安全克制：不提供医疗诊断；疼痛、眩晕或明显不适时提示停止训练。
- 可持续：恢复日也能完成小任务并打卡，避免“没有训练就等于失败”。

本文件故意不记录真实用户的体重、体脂、用药或其他个人健康数据。

## 当前产品状态

### V5：AI 拍照饮食版

确定了“拍照—分析—确认份量与烹饪方式—保存”的饮食记录流程，以及热量和三大营养素估算卡片。

### V6：混合识别版

确定了本地常见餐食模板、历史缓存、复杂餐食再调用云端模型、调用次数限制和免费额度保护的方向。真实云端模型尚未接入当前源码。

### V7：个人健康教练版

已实现网页预览和原生微信小程序。原生版包括：

- 自动 A/B 训练轮换与恢复日。
- 逐组重量、次数、RPE 和 90 秒休息计时。
- 根据完成次数和 RPE 生成下一次重量建议。
- 月度打卡日历、当前/最长连续天数和成就徽章。
- 身高、体重、体脂率、腰围、睡眠、蛋白质和饮水记录。
- 本地规则教练、健康分和最近 7 天报告。
- 饮食照片选择、本地模板估算、份量与烹饪方式修正。
- 本地 JSON 备份与恢复；恢复时对所有嵌套字段做白名单过滤。

## 重要技术决策

- 原生小程序和网页预览器放在同一仓库，但可以独立运行。
- 小程序不使用第三方前端框架，避免构建链和依赖升级成本。
- 健康分和教练建议由纯函数规则引擎生成，当前不依赖收费模型。
- 原生小程序的持久化统一经过 `services/storage.js`，页面不直接定义存储键。
- 训练、日历、营养换算和备份验证尽量保持为纯逻辑，以便用 Node 测试。
- 图片默认不保存、不上传；真实视觉模型必须通过云函数接入，密钥不得进入前端。

详细原因见 [`docs/decisions`](./docs/decisions)。

## 关键路径

- `miniprogram-v7/miniprogram/app.json`：原生页面和底部导航入口。
- `miniprogram-v7/miniprogram/data/defaults.js`：A/B 动作与初始状态。
- `miniprogram-v7/miniprogram/services/storage.js`：本地状态、导出和恢复。
- `miniprogram-v7/miniprogram/utils/validation.js`：健康输入与备份白名单。
- `miniprogram-v7/miniprogram/pages/training/`：逐组训练流程。
- `miniprogram-v7/miniprogram/pages/food/`：照片饮食确认流程。
- `src/versions.mjs`：网页预览中的 V5–V7 版本数据。
- `tests/miniprogram-v7.logic.test.cjs`：原生业务逻辑测试。

## 已知边界

- 真实 Qwen-VL/其他视觉模型尚未接入；饮食页使用透明标注的本地模板演示。
- 当前没有微信云开发、账号系统或跨设备自动同步。
- 当前提醒不在小程序内实现；微信订阅消息需要用户授权和服务端能力。
- 原生页面已通过逻辑、JavaScript、JSON 和工程结构检查，但仍应在微信开发者工具和 iOS/Android 真机上做最终视觉验收。
- `project.config.json` 使用 `touristappid`；真机预览和发布前必须替换为正式 AppID。

## 下一阶段优先级

1. 用微信开发者工具完成首次真机验收，修复设备差异。
2. 配置正式 AppID、主体类目和用户隐私保护指引。
3. 通过微信云函数接入真实图像理解接口，加入超时、限额、删除原图和费用保护。
4. 增加云端加密备份或用户主动同步，保留本地优先模式。
5. 增加动作图示/固定视频，但先确认素材授权和长期可用性。
6. 再考虑身材时间轴；此功能涉及敏感照片，必须先设计明确的本地处理和删除策略。
